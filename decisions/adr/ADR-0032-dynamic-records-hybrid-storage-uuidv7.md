# ADR-0032 — تخزين السجلات الديناميكية: النموذج الهجين (أعمدة نواة + JSONB + ترويج مقنن)

**السياق (Context):** أنواع السجلات في AQL تُعرَّف بالـmetadata عبر الـBuilder (‏OD-BLD-1)، ما يستلزم قراراً معمارياً في تمثيلها التخزيني قبل أول كود منصة: أعمدة لكل حقل، أم EAV، أم JSONB، أم نموذج هجين — مع الحفاظ على م17 (المعرف المرجعي الظاهر ثابت).

**الوضع السابق (Previous state):** ‏ADR-0003 يعتمد PostgreSQL + القواعد الأربع + ‏RLS/FLS؛ ‏م10 يقنن مسار metadata→migration→فحص→اعتماد→تطبيق؛ لا قرار سابق يحدد شكل تخزين الحقول الديناميكية؛ تجربة `aql` تعرّف مرجع السجل المرتبط بأربعة حقول (نوع/معرف/رقم مرجعي/عنوان) `[مصدر: src/mocks/types.ts]`.

**القرار (Decision):**
1. **نموذج هجين**: أعمدة نواة مروَّجة ثابتة لكل السجلات + الحقول الديناميكية في **JSONB**.
2. **أعمدة النواة**: ‏`internal_id` · ‏`reference_number` (الظاهر للمستخدم، **immutable** — م17) · ‏`record_type_id` · مراجع الملكية/الوحدة التنظيمية · ‏`lifecycle_state` · طوابع (`created_at/by` · ‏`updated_at/by`) · تصنيف الحساسية (`classification_level`).
3. **صيغة `internal_id` = UUIDv7** (قابلية فرز زمني + توزيع)؛ **آلية التوليد غير مرتبطة بدالة خاصة بإصدار PostgreSQL معيّن** — يجوز التوليد في طبقة التطبيق أو في قاعدة البيانات، ويُحسم موضع التوليد كقرار تنفيذ لاحق ضمن حدود هذا القرار (قرار مالك، أمر G1 بند 5).
4. فهارس **GIN انتقائية** على JSONB حسب أنماط الاستعلام المقاسة — لا فهرسة شاملة استباقية.
5. **الترويج المقنن**: الحقل كثيف الاستعلام يُرقّى لعمود/فهرس عبر مسار م10 القائم (‏metadata → migration مولَّدة → فحوص → اعتماد → تطبيق) **دون ADR جديد** ما لم يمس حداً معمارياً.
6. هوية السجل تُقنَّن في عقد `contracts/identity/RECORD_IDENTITY.md` (نفس الدفعة) وتحكم كل الربط بين الشاشات والسجلات.
7. لا اشتقاق RLS من الشجرة التنظيمية في v1 — الشجرة للتوجيه/التكليف فقط (‏OD-ORG-1 المقفل).

**البدائل (Alternatives):** أعمدة DDL لكل حقل (مرفوض — انفجار migrations لكل تعديل نوع ويقيد الـBuilder)؛ ‏EAV صرف (مرفوض — استعلامات معقدة وأداء رديء وفقدان أنواع)؛ ‏JSONB صرف بلا أعمدة نواة (مرفوض — يضعف الفهرسة والقيود على الهوية والملكية ودورة الحياة)؛ مخزن وثائقي منفصل (مرفوض — يكسر ADR-0003 ويضاعف مصادر الحقيقة).

**السبب (Rationale):** النواة الثابتة تعطي عقوداً واستعلامات وأمناً (RLS/FLS) مستقرة؛ ‏JSONB يعطي مرونة الـBuilder؛ والترويج المقاس يعالج الأداء عند الحاجة المثبتة لا قبلها — بما يوافق قاعدة Anti-overengineering.

**العواقب (Consequences):** عقود السجلات مستقرة عبر كل الأنواع؛ أهداف قياس الأداء تُعرَّف عند P1 (نطاقات فحص حتى 10–100M سجل كأهداف قياس لا وعود)؛ شاشات Renderer تقرأ النواة مباشرة والديناميكي عبر schema النوع.

**خطة الانتقال (Migration plan):** لا بيانات قائمة للهجرة؛ أول migrations للأساس تمر بمسار مراجعة الـPR المسمى في م10؛ أي ترويج لاحق يمر بمساره المقنن أعلاه.
