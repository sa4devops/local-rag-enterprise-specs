# PHASE 1 PROMPT — Governed Walking Skeleton (Hybrid Lite)
> برومبت مستقل. الصقه في محادثة جديدة مع أي Coding Agent بعد إتمام Phase 0. نفّذ **ما هو مكتوب فقط**. يعتمد على هيكل Phase 0.

---

## السياق (Context)
منصة تشغيل معرفي مؤسسي، شبكة مغلقة، Docker، **منتج واحد قابل للنقل بالإعداد فقط**. النظام **metadata-driven**: الشاشات والحقول تُعرَّف كبيانات في `platform_db`، وتُبنى من لوحة الأدمن. **الحوكمة قبل القدرة**: الصلاحية تُفحص قبل أي عملية بيانات، وكل تغيير يُسجَّل في Audit Log. الواجهة **مؤسسية خاصة** (Arabic/English, RTL). الـ LLM ليس مصدر الحقيقة، ولا LLM في هذه المرحلة أصلاً.

## هدف المرحلة (Goal)
بناء **شريحة رأسية رفيعة تمرّ خلال كل الطبقات الحرجة** لإثبات جوهر المشروع تحت الحوكمة end-to-end: هوية حقيقية + تنظيم + بوابة صلاحية مُنفَّذة + سجل ميتاداتا + تدقيق + **شريحة باني الشاشات الكاملة** (تعريف شاشة → توليد migration → مراجعة/اعتماد → تطبيق → CRUD). الأساس **حقيقي لكن حدّه الأدنى** بحيث **لا يُعاد بناء أي شيء لاحقاً**.

## داخل النطاق (In Scope)

### 1) Local Auth حقيقي (مع تجريد للـ IdP)
- تسجيل دخول/خروج، تجزئة كلمات مرور، جلسة/JWT.
- تنفيذ `LocalAuthProvider` خلف واجهة `IdentityProvider` (المعرَّفة في Phase 0) — بحيث يُضاف LDAP/AD/OIDC لاحقاً **كتنفيذ جديد بلا تعديل جوهر**.
- **admin واحد مزروع** (seed).
- الرقم الوظيفي واسم المستخدم **ثابتان ومحميان** (immutable على مستوى القاعدة).
- دعم تعطيل/تفعيل المستخدم، وتغيير كلمة المرور.

### 2) النموذج التنظيمي (Organization)
- كيانات: `organization` → `sector` → `department`.
- ربط المستخدم بإدارة/قطاع.
- **2–3 أدوار مزروعة**: `SystemAdmin`, `Manager`, `User`.

### 3) العمود الفقري للصلاحيات (Permission Spine — مُنفَّذ لا وهمي)
- `PermissionService` **يُستدعى فعلياً قبل كل عملية بيانات** (قراءة/كتابة).
- قواعد v1 مبسّطة لكنها حقيقية: **دور + ملكية (ownership) + تصنيف (classification)** + **default-deny**.
- كل سجل أعمال يحمل: `department_id`, `sector_id`, `owner_user_id`, `classification` (enum: Public/Internal/Confidential/Secret/Restricted) + حقول تدقيق (created_by/at, updated_by/at).
- بوابة صلاحية مركزية تُرجع قراراً (allow/deny) + السبب/السياسة.

### 4) سجل الميتاداتا (Metadata Registry في platform_db)
- جداول: `screen_definitions`, `field_definitions`, `app_registry` (اسم التطبيق/الـ schema)، `migration_records` (نسخة/حالة/توقيت/منفّذ).
- باني الشاشات يكتب هنا؛ لا شاشات مضمّنة في الكود.

### 5) العمود الفقري للتدقيق (Audit Spine)
- `audit_log` **append-only** في platform_db.
- `AuditService` **تستدعيه كل عملية تغيير** (من/ماذا/متى/على أي مورد/قرار الصلاحية).

### 6) شريحة باني الشاشات الرأسية (Screen Builder Vertical Slice) — القلب
تدفق كامل من لوحة الأدمن:
1. **تعريف شاشة + حقول** (الاسم، النوع، الإلزامية، التصنيف) → تُحفظ في سجل الميتاداتا.
2. **توليد ملف migration** لجدول في `apps_db` تحت **schema خاص بالتطبيق** (schema-per-app). **ملف migration فقط — لا DDL مباشر من أي نموذج.**
3. **فحص تعارض** (أسماء/أنواع/مفاتيح مكرّرة) + **فحص مخاطر** (تصنيف/كسر توافق).
4. **إجراء اعتماد** من الأدمن (approve/reject) — قبل التطبيق.
5. **تطبيق الترحيل** داخل schema التطبيق + **versioning** لكل migration.
6. **سجل توثيق التوليد** (شاشة تُظهر: ماذا وُلّد/عُدّل/أُوقف/حُذف، ومتى ومن).
7. **شاشة CRUD أساسية** للمستخدم على الكيان المولَّد.
> **كل خطوة**: تمرّ عبر `PermissionService` أولاً، وتُسجَّل في `AuditService`.

### 7) قشرة Enterprise UI مصغّرة
- Arabic/English + **RTL كامل**.
- تستضيف: صفحة الدخول، **Admin Console** (المستخدمون/التنظيم/الأدوار + باني الشاشات + مراجعة/اعتماد التوليد + توثيق التوليد)، و**عارض الشاشة المولَّدة** مع CRUD.
- بلا دردشة/متجر/باني workflow.

## خارج النطاق صراحةً (Out of Scope)
RAG/OCR/ملفات، الوكلاء وبوابة النماذج وأي LLM، سير الاعتماد الكامل والإشعارات، RBAC/ABAC/ReBAC الكامل وField-Level المتقدّم، المتجر والتكاملات، التقارير الطويلة، المراقبة الكاملة، الفصل الفيزيائي للقواعد.

## القيود التقنية (Constraints)
- كل شيء **configuration-driven** و**قابل للنقل** (نفس كود Phase 0، إعدادات فقط للإنتاج).
- Backend **عديم الحالة**؛ الجلسات في Redis/JWT.
- **لا DDL مباشر من نموذج**؛ فقط ملفات migration مُراجَعة ومعتمدة ومؤرشَفة.
- **default-deny** في كل قرار صلاحية؛ الصلاحية **قبل** أي جلب بيانات.
- تكامل schema-per-app جاهز لدعم الفصل الفيزيائي لاحقاً عبر تغيير endpoint في الإعداد (بلا افتراض قاعدة واحدة في الكود).

## معايير القبول (Acceptance Criteria)
- admin يسجّل الدخول → ينشئ organization/sector/department + مستخدماً بدور.
- الرقم الوظيفي واسم المستخدم لا يمكن تعديلهما (يُرفض على مستوى القاعدة).
- admin يعرّف شاشة + حقول → يولّد migration → يظهر فحص التعارض/المخاطر → يعتمد → يُطبَّق الترحيل → يظهر في سجل التوثيق.
- تُنشأ شاشة CRUD تعمل على الكيان المولَّد، وكل عملية عليها تمرّ ببوابة الصلاحية وتُسجَّل في Audit Log.
- محاولة عملية بلا صلاحية → **تُمنع** وتُسجَّل.
- الواجهة تعمل بالعربية والإنجليزية مع RTL صحيح.
- لا شيء خارج الـ scope؛ لا LLM؛ لا DDL مباشر.

## تعليمات للـ Coding Agent
نفّذ الـ scope حرفياً. اجعل الأساس حقيقياً لكن حدّه الأدنى بحيث لا يحتاج إعادة بناء عند إضافة RBAC/ABAC/ReBAC والوكلاء لاحقاً. عند أي غموض، **اسأل قبل الافتراض**. سجّل القرارات في `specs/phase-1/notes.md`.
