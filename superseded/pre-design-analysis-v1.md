# تحليل ما قبل التصميم
## منصة تشغيل معرفي مؤسسي تعمل في شبكة مغلقة — Air-Gapped Enterprise Knowledge / RAG Platform

**الحالة:** مرحلة *ما قبل التصميم* (Pre-Design).
**الغرض:** تثبيت الفهم المشترك، كشف التعارضات والمخاطر مع توصيات عملية لحلّها، وتحديد الأسئلة الحاجزة التي يجب حسمها **قبل** الانتقال إلى التصميم التفصيلي (الأقسام العشرين) أو أي كود.

**لماذا لم ننتقل إلى التصميم الكامل بعد:** التزاماً بقاعدتك الأساسية — «إذا وُجد نقص أو غموض أو تعارض، اسأل قبل التنفيذ، ولا تبنِ افتراضات غير مُوافَق عليها». توجد قرارات حاجزة حقيقية (مواصفات سيرفر الإنتاج، قرار علامة Open WebUI، تحديد الأنظمة الداخلية مقابل الخارجية، مقياس البيانات، مزوّد الهوية) لا يصحّ تثبيت تصميم نهائي قبل حسمها، وإلا تحوّل التصميم إلى افتراضات غير مُعتمدة.

## 1) الفهم المُجمَّع — مبادئ مثبتة (تُعامَل كأساس غير قابل للتفاوض)

هذه نقاط استخلصتها من وثيقة التأسيس و`database` و`سياسة الكاش` و`OKF` وبقية المرفقات. أعتبرها **ثوابت التصميم**:

**1.1 طبيعة النظام:** منصة تشغيل مؤسسي (Platform)، وليست chatbot. الواجهة الحوارية وسيلة، والجوهر هو حوكمة المعرفة والعمليات.

**1.2 مصدر الحقيقة (Source of Truth):** النموذج اللغوي **ليس** مصدر الحقيقة. الحقيقة = قاعدة بيانات PostgreSQL + المستندات المعتمدة بعد الفهرسة والتحقق + الـ APIs الداخلية الموثوقة + السجلات الرسمية. الـ LLM **يعالج** البيانات التي يجلبها له الـ Agent، ولا يُجيب من ذاكرته العامة في بيانات الأعمال.

**1.3 ترتيب الصلاحيات قبل الاسترجاع:** تُفحص الصلاحيات **قبل** أي retrieval أو SQL أو استدعاء API، لا بعده. لا استرجاع لبيانات لا يملك المستخدم صلاحيتها أصلاً.

**1.4 الحوكمة والتدقيق:** كل عملية (قراءة حرجة، إضافة، تعديل، حذف، تصدير، إرسال، اعتماد، تشغيل workflow، استخدام AI Action) تُسجَّل في **Audit Log**.

**1.5 الكاش (سياسة الكاش المعتمدة بتعديلاتك):**
- قاعدة البيانات هي مصدر الحقيقة الأول؛ المستندات Evidence/Approved Source بحسب تصنيفها.
- **final response cache مُعطَّل افتراضياً** في بيانات الأعمال.
- يُسمح بـ RAG cache لـ (retrieval / embeddings / reranking) فقط، **مربوطاً** بنسخة المستند + نسخة الفهرس + نسخة السكيم + نطاق الصلاحيات + الطابع الزمني.
- إعادة تحقّق **live** قبل أي تعديل أو موافقة أو تصدير رسمي أو فحص صلاحيات.
- KV Cache مسؤولية طبقة model serving، لا application cache.

**1.6 النظام metadata-driven:** أهم قرار معماري. الشاشات والحقول والصلاحيات والروابط تُعرَّف كبيانات (metadata) في قاعدة المنصة، لا ككود ثابت. هذا ما يجعل بناء الشاشات من لوحة الأدمن ممكناً دون تعديل كود.

**1.7 المعمارية الطبقية والوكلاء (Agents):**
المسار: `Backend API` ← `Orchestrator/Router` ← `Specialized Agents` ← `مصادر البيانات` ← `Validation` ← `Answer/Reasoning` ← `Audit`.
الوكلاء: Orchestrator، Retrieval، SQL (للقراءة فقط، مقيّد بالسكيم والصلاحيات)، Validation، Reasoning، Security/Policy، OCR/Doc، API-Integration، Report، Admin/Metadata.

**1.8 نموذج الصلاحيات المؤسسي:**
`RBAC` (الأدوار) + `ABAC` (الصفات/السياق) + `ReBAC` (العلاقات الهرمية) + `Data Classification` (Public / Internal / Confidential / Secret / Restricted) + صلاحيات على مستوى الحقل/السجل/الشاشة/العملية + Least Privilege + Need-to-know + Separation of Duties (SoD).

**1.9 الثوابت المحمية (Immutable):** الرقم المرجعي لكل شاشة (يختلف بين الشاشات والإدارات)، والرقم الوظيفي للمستخدم، وبيانات المستخدم الأساسية (الاسم/الرقم الوظيفي). يُمنع التلاعب بها بتصميم قاعدة البيانات نفسها.

**1.10 دورة حياة الـ Workflow والاعتماد:** يدعم النظام حالتين:
1. workflow مبني على ربط سابق قام به الأدمن بين الشاشات/الإدارات.
2. workflow ينشئه المستخدم بنفسه، فيمرّ على: إنشاء → مسودة → فحص تلقائي للتعارضات والصلاحيات والمخاطر → طلب اعتماد → مراجعة Manager/مدير النظام → قبول/رفض/طلب تعديل → تفعيل → تسجيل كل خطوة في Audit Log.

**1.11 الميزات الوظيفية الكبرى:** Tasks/قوالب، Projects (تجميع المحادثات والملفات حسب الصلاحية)، شاشة إشعارات تفاعلية (دردشة على كل إشعار)، Store/متجر برامج المؤسسة للمستخدم النهائي، شاشة أدمن لإدارة الربط الفني (MCP/API) منفصلة عن المتجر، شاشة About (الأدوات المفتوحة وتراخيصها والـ Attribution).

**1.12 التشغيل المغلق:** يعمل offline داخل Docker / Docker Compose، قابل للنقل إلى سيرفر آخر داخل شبكة مغلقة، مع آلية تحقق من نجاح التشغيل.

**1.13 مخازن البيانات والنماذج المرشحة:**
- `platform_db` (الميتاداتا) + `apps_db` (schema-per-app) + `integration_db` + `reporting_db`.
- `Qdrant` (متجهات) + `OpenSearch` (full-text) + `MinIO` (ملفات) + `Redis` (cache/queue) + `Neo4j` (graph، اختياري).
- Runtimes: `Ollama` / `vLLM`. نماذج مرشحة: Qwen3 / Qwen3-Coder 30B، Qwen2.5-VL-32B (vision)، nomic-embed-text-v1.5 (embedding)، bge-reranker-v2-m3 (reranking)، Whisper (تفريغ صوتي).

## 2) حقائق متحقَّق منها خارجياً (Verified External Facts)

تحقّقت من النقاط التالية عبر بحث ويب لأنها تؤثر مباشرة على قرارات معمارية وترخيصية:

**2.1 OKF — Open Knowledge Format (من Google Cloud):** حقيقي، نُشر 12 يونيو 2026، إصدار v0.1 (Draft)، رخصة Apache 2.0. هو **صيغة** (ملفات Markdown + YAML frontmatter)، **وليس منصة** → صديق للشبكة المغلقة. لكن **المُولِّد المرجعي** (reference producer) يعتمد على BigQuery + Gemini سحابياً → **غير قابل للاستخدام داخل الشبكة المغلقة**؛ يجب توليد الحزم بأدوات محلية. **التوصية:** استخدامه كطبقة *فهرس معرفي / قاموس بيانات (data dictionary) / مسرد مصطلحات / تعليمات Agents* فوق قاعدة البيانات، **لا** بديلاً عن قاعدة البيانات أو الصلاحيات أو التدقيق. (تحليلك في `OKF.docx` كان صحيحاً.)

**2.2 ترخيص Open WebUI — ⚠️ نقطة حرجة:** الإصدارات v0.6.6 فما فوق تحتوي بند حماية للعلامة (branding protection). **لا يمكن** إزالة اسم/شعار "Open WebUI" من الواجهة **إلا** إذا تحقّق أحد الشروط: ≤ 50 مستخدماً (خلال 30 يوماً متحركة)، أو ترخيص enterprise، أو كونك مساهماً رسمياً معتمداً. يمكن عمل fork من الإصدار v0.6.5 (رخصة BSD-3 بلا هذا القيد) لكنك ترث عبء الصيانة. هذا **حرجٌ** لمتطلب White-Label لديك، خاصة مع ~30 مستخدماً الآن ونيّة التوسع لاحقاً (قد تتجاوز 50).

**2.3 PentAGI (vxcontrol/pentagi):** حقيقي، رخصة MIT. منصة pentest متعددة الوكلاء (orchestrator + researcher + developer + executor)، بيئة Docker معزولة، Postgres + pgvector، Neo4j graph، تدعم Ollama/vLLM (قابلة للعمل مغلقاً). **نستعير الأنماط لا شكل المنتج.**

**2.4 RedAmon (samugit83/redamon):** حقيقي، رخصة MIT. إطار red-team بالذكاء الاصطناعي. يدعم اختيار provider/model لكل مشروع من الإعدادات (يطابق متطلبك «تغيير النماذج دون كود»)، جلب ديناميكي للنماذج، بوابات تأكيد بشرية (human-in-the-loop) قبل تنفيذ الأدوات (تطابق بوابات الاعتماد لديك)، ذاكرة Neo4j، إعداد config-driven عبر Postgres/Prisma، يعمل offline. **نستعير الأنماط.**
> ملاحظة: PentAGI وRedAmon أداتا أمن **هجومي** → الأنسب اعتبارهما **تكاملاً مستقبلياً محتملاً** (كأدوات MCP/API) لفريق Red Team/CERT، لا أساساً للمنصة المؤسسية العامة.

**2.5 «Spacekit من Microsoft» = GitHub Spec Kit (github/spec-kit):** أداة `specify` CLI، رخصة MIT. سير العمل: `constitution → specify → clarify → plan → tasks → implement`. قواعدك «غير القابلة للتفاوض» → ملف `constitution.md`. كل وحدة/مرحلة → spec branch مستقل (يُمكِّن أسلوبك «برومبت مستقل لكل مرحلة في محادثة جديدة»).

## 3) التعارضات والمخاطر + التوصيات (Conflicts / Risks + Recommendations)

**التعارض 1 — نموذج قاعدة البيانات:**
البرومبت يطلب «قاعدة منفصلة لكل إدارة/قسم + توليد تلقائي»، بينما `database.docx` (وهو الأعلى جودة) يعتمد **schema-per-app + RLS** للعزل التنظيمي.
**التوصية:** اعتماد نموذج `database.docx`. أن يكون «التوليد التلقائي» = **migrations مُراجَعة ومُؤرشَفة عبر بوابة اعتماد**، لا تنفيذ DDL مباشر من الـ LLM على الإنتاج. الفصل الفيزيائي لقاعدة مستقلة يحدث **فقط** عند نمو التطبيق أو حساسيته (وليس افتراضياً لكل إدارة).

**التعارض 2 — علامة Open WebUI (مرتبط بـ 2.2):**
متطلب White-Label يصطدم بترخيص v0.6.6+.
**الخيارات:** (أ) ترخيص enterprise، (ب) fork من v0.6.5، (ج) **بناء واجهة مؤسسية خاصة تكون «وجه» العلامة، وإبقاء Open WebUI كسطح دردشة داخلي** — *موصى*، (د) إبقاء علامة Open WebUI كما هي.

**التعارض 3 — «الواجهة ستكون Open WebUI» مقابل «Open WebUI مجرد طبقة»:**
Open WebUI لا تستضيف الشاشات الديناميكية ولا المتجر ولا باني الـ workflow ولا الإشعارات ولا لوحات الأدمن.
**التوصية:** Open WebUI = **سطح الدردشة فقط**؛ وواجهة مؤسسية منفصلة (Next.js مثلاً) لكل الباقي. (لاحظ أنك نفسك خفّفت العبارة في البرومبت بقولك إنها «ليست بالضرورة قلب النظام».)

**المخاطرة 4 — العتاد والتزامن:**
عتاد التطوير (RTX 5090 Laptop، 24GB VRAM، 64GB RAM) **لا** يشغّل reasoning + coder + vision + embed + rerank **معاً**؛ Ollama يبدّل نموذجاً رئيسياً واحداً في الذاكرة. الخدمة المتزامنة لـ ~30 مستخدماً تحتاج **سيرفرات إنتاج** (المواصفات مطلوبة)، وغالباً **vLLM** أنسب للتزامن من Ollama.

**المخاطرة 5 — فهم الفيديو offline:** ثقيل جداً على عتاد متوسط.
**التوصية:** حصر الإصدار الأول v1 في **تفريغ الصوت (Whisper) + وصف لقطات مفتاحية (keyframe captioning)**؛ وتأجيل الفهم البصري الكامل للفيديو (**مع عدم حذفه** من التصميم).

**المخاطرة 6 — المتجر و OAuth لكل مستخدم:** يعمل فقط للأنظمة المتاحة على الشبكة الداخلية. الأنظمة الخارجية (SaaS عبر الإنترنت) **غير قابلة للوصول** داخل بيئة مغلقة.
**التوصية:** تحديد أي الأنظمة داخلية (قابلة للربط) مقابل خارجية (غير قابلة للوصول مغلقاً) — سؤال حاجز رقم 4.

## 4) المعمارية المقترحة (Proposed Architecture)

الطبقات (انظر المخطط المرئي المرفق `معمارية_النظام.svg`):

1. **Presentation:** Open WebUI (سطح الدردشة) + Enterprise UI (الشاشات · المتجر · باني الـ Workflow · الإشعارات · الأدمن).
2. **Backend API (FastAPI):** AuthN/Z · RBAC/ABAC/ReBAC · Dynamic Screens · Records · Audit · License.
3. **Agent Orchestrator / Router:** Request Classifier · Context Builder · **Permission Gate** · **No-Guessing Guard**.
4. **Agents:** Retrieval · SQL (read-only) · Validation · Reasoning · Security/Policy · OCR/Doc · API-Integration · Report · Admin/Metadata.
5. **Knowledge & Models:** طبقة OKF (فهرس/مسرد/تعليمات) + LLM Gateway → (Ollama/vLLM: Reasoning · Coder · Vision · Embed · Rerank).
6. **Sources of Truth:** platform_db · apps_db (schema-per-app) · integration_db · reporting_db · Qdrant · OpenSearch · MinIO · Redis · Neo4j(opt).
7. **Connectors:** MCP · API · Webhook · DB Connector → الأنظمة المؤسسية الداخلية (بريد/موارد بشرية/مالية/...).
- **شريط عرضي شامل (Cross-cutting):** Audit Log · Cache (تسريع فقط) · Monitoring (Prometheus/Grafana/Loki) · License · Hardware Manager · Security/Policy.

**تدفق الطلب (Request Flow) خطوة بخطوة:**
طلب المستخدم → Backend API (مصادقة) → Orchestrator (تصنيف الطلب + بناء السياق + **فحص الصلاحية قبل أي جلب**) → الوكيل المناسب يجلب البيانات **بحدود الصلاحية** (Retrieval/SQL/API/OCR) → Validation Agent يتحقق → الـ LLM **يعالج فقط** ما جلبه الوكيل (تلخيص/تحليل/تقرير) مع **No-Guessing Guard** → عرض الإجابة **مع المصدر** → تسجيل في Audit. عند أي عملية حرجة (تعديل/موافقة/تصدير): **إعادة تحقق live** من المصدر قبل التنفيذ.

## 5) خطة التنفيذ المقترحة (8 مراحل) — متوافقة مع ترتيب وثيقة التأسيس

- **Phase 0 — الأساس:** repo + تهيئة Spec Kit + `constitution.md` + هيكل Docker Compose.
- **Phase 1 (MVP — الوحدة 1):** Auth + الهيكل التنظيمي (Organization) + RBAC أساسي + الثوابت المحمية (الرقم الوظيفي/المرجعي immutable).
- **Phase 2:** باني الشاشات/الحقول الديناميكي + محرك السجلات (migrations مُراجَعة إلى apps_db schema-per-app + **شاشة توثيق التوليد** + RLS).
- **Phase 3:** المسودة/الاعتماد/التدقيق (Draft/Approval/Audit) + أزرار الإجراءات (Action Buttons ككيانات محكومة بالصلاحية).
- **Phase 4:** ملف الكيان (Entity Profile) + RBAC/ABAC/ReBAC الكامل + صلاحيات الحقل/السجل.
- **Phase 5 (MVP — الوحدتان 4–5):** الملفات/OCR/RAG + SQL Agent الآمن + طبقة OKF + No-Guessing Guard.
- **Phase 6:** جسر Open WebUI + LLM Gateway + اختيار النموذج/المزوّد لكل وكيل (نمط RedAmon) + Hardware/Runtime Manager.
- **Phase 7:** موصلات API + شاشة الأدمن للربط الفني + المتجر المحكوم بالصلاحيات + MCP.
- **Phase 8:** التقارير/التصدير (تقارير طويلة قسماً قسماً) + المراقبة (Prometheus/Grafana/Loki) + حزمة النقل المغلق (images/wheels/models + استعادة/تحقق).

**مؤجَّل (غير محذوف):** فهم الفيديو الكامل، فصل التطبيقات الكبيرة إلى قواعد مستقلة، تكامل PentAGI/RedAmon كأدوات.

## 6) طريقة العمل (Working Method)

- Spec Kit: `constitution.md` يحمل القواعد الثابتة؛ كل مرحلة = spec/plan/tasks في فرع مستقل.
- **برومبت مستقل لكل مرحلة** يُنفَّذ في **محادثة جديدة** (لتفادي التشتت في المحادثة الرئيسية).
- بعد حسم الأسئلة الحاجزة → أُسلّم **حزمة التصميم**: مخططات (Context / Data-Flow / Agent-Design / ERD) + نموذج الصلاحيات التفصيلي + عقود الـ API + سكيمة الميتاداتا + Docker Compose + `constitution.md`، ثم نبدأ Phase 1 ببرومبتها المستقل.

## 7) الأسئلة الحاجزة (10) — مطلوب حسمها قبل التصميم التفصيلي

1. **مواصفات سيرفر الإنتاج** (نوع/عدد GPU وVRAM، RAM، CPU، التخزين) + **العدد المتزامن** المستهدف للمستخدمين. *(يحكم تصميم التزامن واختيار Ollama مقابل vLLM.)*
2. **قرار علامة Open WebUI:** (أ) ترخيص enterprise، (ب) fork v0.6.5، (ج) واجهة مؤسسية خاصة + Open WebUI داخلي *(موصى)*، (د) إبقاء العلامة. ⭐
3. **اعتماد نموذج `database.docx`** (schema-per-app + RLS) مع «التوليد التلقائي = migrations عبر بوابة اعتماد»؟ ⭐
4. **أي الأنظمة سنربطها**، وأيها **داخلي** (قابل للربط مغلقاً) مقابل **خارجي SaaS** (غير قابل للوصول)؟
5. **مقياس البيانات:** عدد الإدارات/الأقسام، عدد التطبيقات/الشاشات، أحجام السجلات، حجم مكتبة المستندات.
6. **OKF الآن أم لاحقاً؟** ⭐
7. **تأكيد GitHub Spec Kit** + **أي وكيل ذكاء سيكتب الكود** (Claude Code / Copilot / Codex / Cursor)؟
8. **الهوية:** هل لديكم IdP داخلي (LDAP/AD/Keycloak/OIDC) أم نبني مصادقة محلية؟
9. **اللغات وRTL** المطلوبة في الواجهة المؤسسية.
10. **تأكيد بدء MVP بالوحدة 1 (Auth + Org)** + **تأكيد نهائية تصنيف البيانات** (Public/Internal/Confidential/Secret/Restricted).

**القرارات الثلاثة ذات الأولوية ⭐:** نموذج قاعدة البيانات (3) · علامة Open WebUI (2) · OKF الآن/لاحقاً (6).
> ملاحظة: السؤال (1) أيضاً على المسار الحرج لأنه يحدد تصميم التزامن وخدمة النماذج.
