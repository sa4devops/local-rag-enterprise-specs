## §1 الغرض والنطاق
يعرّف هذا الملف كيف يعمل أي AI Coding Agent داخل المشروع من لحظة استلام مهمة حتى تسليمها: دورة التشغيل، مصادر الحقيقة، المحظورات الصلبة، وقالب بطاقة المهمة الإلزامي. لا يضيف قواعد جديدة فوق الدستور والمنهجية — بل يشغّلها.

## §3 مصادر الحقيقة وترتيبها
1. `platform/docs/SPEC_SOURCE.md` أولاً — يحدد إصدار المواصفات النافذ.
2. محتوى إصدار specs المشار إليه **كاملاً** — المواصفات هي مصدر الحقيقة الوحيد.
3. سلسلة م20 عند التعارض. الغموض ⇒ سؤال Clarify، لا افتراض ولا استنتاج.

## §4 المحظورات الصلبة (تشغيل لا إلغاء لتفصيل §7 في مراجعة ما-قبل-التنفيذ)
- **D9 كما هو:** لا أسرار ولا Base URL ولا API Key ولا endpoints حقيقية في أي UI أو DTO موجَّه للواجهة؛ `secret_ref` بالاسم فقط.
- **لا Terminal UI** ولا أي واجهة/ميزة سايبرية بأي صيغة.
- **لا كود من PentAGI أو RedAmon**؛ وOpen WebUI عبر مسار D8 فقط وبلا نسخ UI.
- **لا تنفيذ أي Phase** أو ملف تنفيذ بلا بطاقة مهمة معتمدة صراحة.
- تعديل الملفات القائمة **بنمط Patch/Additive حصراً** — لا إعادة كتابة كاملة ولا حذف قرارات.
- الالتزام بالتسمية المقفلة (`runs.list`/`runs.detail`؛ كتالوج الأفعال المغلق + run/escalate/compare) وبكل OD كما هو مسجّل — الـ OD ليس دعوة للاجتهاد.

## §5 قاعدة المراجع الأمامية
تصاميم المراحل P1–P8 تُؤلَّف **عند تفعيل كل مرحلة** (phase-roadmap). معرّفات FR-x الواردة في مواصفات UI (مثل FR-6.1) مراجعُ أمامية تُحسم نصاً عند تأليف تصميم مرحلتها — يُمنع تخمين محتواها أو البناء عليه قبل ذلك. «تصاميم v0.3 المستعادة في phases/designs/ هي المدخل الإلزامي والموطن التعريفي لمعرفات FR؛ ويبقى «تصميم التفعيل» المنتَج عند تفعيل المرحلة هو النص النافذ.»

## §6 قالب بطاقة المهمة (Task Card) — إلزامي قبل أي كود
| الحقل | الوصف |
|---|---|
| task_id | معرّف فريد ASCII |
| المرحلة | Phase المستهدفة ومرجع تصميمها |
| الهدف | جملة نتيجة واحدة قابلة للفحص |
| الملفات ضمن النطاق | قائمة صريحة مغلقة |
| خارج النطاق صراحة | ما يُمنع لمسه |
| القيود الحاكمة | D9 · مواد الدستور ذات الصلة · ODs المفتوحة المعنية |
| معايير القبول | بنود PASS/FAIL قابلة للأتمتة ما أمكن |
| بوابة الفحص | ملف البوابة/القسم الذي سيُشغَّل |
| اعتماد المالك | مرجع موافقة SA + التاريخ |
**قاعدة:** بلا بطاقة معتمدة من SA لا يبدأ تنفيذ — ولا استثناء «مهمة صغيرة».

## §8 مخرجات الوكيل
كل تسليم = التغييرات + تقرير فحص ذاتي (نقاء/حدود/نطاق/بوابة) + إدخال Handoff مقترح. **لا commits ولا push ولا وسوم ولا releases من أي وكيل** — كلها بيد SA حصراً.

## §10 سلّم الجاهزية (Readiness Ladder) — ست درجات
| الدرجة | تعريف سطري |
|---|---|
| **Architecture-ready** | الحدود والعقود المتوقعة والتبعيات ومرحلة التفعيل موثقة — لا كود |
| **Contract-ready** | العقود/الواجهات معلنة ومجمّدة الشكل (schemas/interfaces) دون تنفيذ فعلي |
| **Provider-ready** | التجريد موجود وتنفيذٌ افتراضي أدنى يعمل خلفه (memory/fs/PG) |
| **Implementation-ready** | تصميم المرحلة معتمد وبطاقات المهام جاهزة للبدء |
| **Runtime-activated** | القدرة مفعّلة فعلاً في بيئة التشغيل بعلَمها وإعدادها |
| **Production-scaled** | مفعّلة بإعدادات إنتاج مقاسة (أحجام/نسخ/موارد بقياس فعلي) |

## §11 قاعدتا الحراسة (Guard Rules) — (v1.0-architecture-baseline)
**1) قاعدة منع الحذف (Anti-deletion — م22):** Do not implement a future capability before its phase. Do not remove its architectural place merely because it is deferred. — لا تُبنى قدرة مستقبلية قبل مرحلتها، ولا يُحذف موطؤها المعماري (الحدود/العقود/التبعيات الموثقة) لمجرد تأجيلها.
**2) قاعدة منع الهندسة الزائدة (Anti-overengineering):** لا يُبنى abstraction مبكر إلا بتحقق شرط واحد على الأقل: (أ) تستخدمه المرحلة الحالية فعلاً؛ أو (ب) أكثر من تنفيذ واحد **متوقع في تصاميم معتمدة**؛ أو (ج) التقنية يصعب استبدالها لاحقاً دون تجريد؛ أو (د) عقد عام يربط أكثر من واجهة. القدرات البعيدة تكتفي بـ: حدود موثقة · عقود متوقعة · تبعيات · مرحلة تفعيل · قرارات مؤجلة مسماة.
**توضيح القيم التشغيلية:** أعداد النسخ (replicas) · أحجام النماذج · مدد الاحتفاظ النهائية · أحجام التخزين · حدود التزامن · تحجيم GPU للإنتاج — **قيم تشغيلية تُضبط بالقياس لكل نشر، لا قرارات معمارية مفتوحة** (فهرسها في `decisions/open-decisions.md`).
