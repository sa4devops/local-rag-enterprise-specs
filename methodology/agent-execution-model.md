# agent-execution-model.md — نموذج تشغيل وكلاء التنفيذ (Coding Agents)

> **Version:** 1.0 — Proposed (Post-merge Fix Batch) · **Date:** 2026-07-08 · **الموضع:** `methodology/agent-execution-model.md`
> **Authority:** يعمل تحت سلسلة المرجعية (constitution م20): constitution ← Methodology ← Catalog ← Open-Decisions ← phase-roadmap ← تصاميم المراحل ← المواصفات التفصيلية. هذا الملف هو «النسخة المرجعية الموسّعة» لنموذج تشغيل الوكلاء التي تشير إليها المنهجية، ويُقرأ دائماً مع `methodology/PRE_IMPLEMENTATION_CONSOLIDATION_REVIEW.md` §7 و`coding-standards.md` و`testing-strategy.md`.

## §1 الغرض والنطاق
يعرّف هذا الملف كيف يعمل أي AI Coding Agent داخل المشروع من لحظة استلام مهمة حتى تسليمها: دورة التشغيل، مصادر الحقيقة، المحظورات الصلبة، وقالب بطاقة المهمة الإلزامي. لا يضيف قواعد جديدة فوق الدستور والمنهجية — بل يشغّلها.

## §2 دورة التشغيل القياسية
| # | الخطوة | الخلاصة |
|---|---|---|
| 1 | Clarify | أسئلة الغموض تُطرح وتُحسم قبل أي عمل — لا افتراضات |
| 2 | Task Card | بطاقة مهمة معتمدة من SA وفق §6 — شرط البدء |
| 3 | قراءة المصادر | وفق ترتيب §3 كاملاً قبل أول تعديل |
| 4 | التنفيذ | Patch/Additive حصراً على القائم؛ ملفات جديدة فقط إن نصّت البطاقة |
| 5 | فحوص ذاتية | نقاء (لا أسيجة كود/لا أوامر) · حدود D9 · تسمية مقفلة · نطاق البطاقة |
| 6 | البوابة | تشغيل بنود بوابة الحزمة/المرحلة المعنية وتوثيق النتائج |
| 7 | Handoff | تقرير تسليم + إدخال handoff مقترح — الدمج/الرفع بيد SA حصراً |

## §3 مصادر الحقيقة وترتيبها
1. `platform/docs/SPEC_SOURCE.md` أولاً — يحدد إصدار المواصفات النافذ.
2. محتوى إصدار specs المشار إليه **كاملاً** — المواصفات هي مصدر الحقيقة الوحيد.
3. سلسلة م20 عند التعارض. الغموض ⇒ سؤال Clarify، لا افتراض ولا استنتاج.

## §4 المحظورات الصلبة (تشغيل لا إلغاء لتفصيل §7 في مراجعة ما-قبل-التنفيذ)
- **D9 كما هو:** لا أسرار ولا Base URL ولا API Key ولا endpoints حقيقية في أي UI أو DTO موجَّه للواجهة؛ `secret_ref` بالاسم فقط.
- **لا Terminal UI** ولا أي واجهة/ميزة سايبرية بأي صيغة.
- **لا كود من PentAGI أو RedAmon**؛ وOpen WebUI عبر مسار D8 فقط وبلا نسخ UI.
- **لا تنفيذ أي Phase** أو ملف تنفيذ بلا بطاقة مهمة معتمدة صراحة.
- تعديل الملفات القائمة **بنمط Patch/Additive حصراً** — لا إعادة كتابة كاملة ولا حذف قرارات.
- الالتزام بالتسمية المقفلة (`runs.list`/`runs.detail`؛ كتالوج الأفعال المغلق + run/escalate/compare) وبكل OD كما هو مسجّل — الـ OD ليس دعوة للاجتهاد.

## §5 قاعدة المراجع الأمامية
تصاميم المراحل P1–P8 تُؤلَّف **عند تفعيل كل مرحلة** (phase-roadmap). معرّفات FR-x الواردة في مواصفات UI (مثل FR-6.1) مراجعُ أمامية تُحسم نصاً عند تأليف تصميم مرحلتها — يُمنع تخمين محتواها أو البناء عليه قبل ذلك.

## §6 قالب بطاقة المهمة (Task Card) — إلزامي قبل أي كود
| الحقل | الوصف |
|---|---|
| task_id | معرّف فريد ASCII |
| المرحلة | Phase المستهدفة ومرجع تصميمها |
| الهدف | جملة نتيجة واحدة قابلة للفحص |
| الملفات ضمن النطاق | قائمة صريحة مغلقة |
| خارج النطاق صراحة | ما يُمنع لمسه |
| القيود الحاكمة | D9 · مواد الدستور ذات الصلة · ODs المفتوحة المعنية |
| معايير القبول | بنود PASS/FAIL قابلة للأتمتة ما أمكن |
| بوابة الفحص | ملف البوابة/القسم الذي سيُشغَّل |
| اعتماد المالك | مرجع موافقة SA + التاريخ |
**قاعدة:** بلا بطاقة معتمدة من SA لا يبدأ تنفيذ — ولا استثناء «مهمة صغيرة».

## §7 الإحالة الملزمة
التعليمات السبع التفصيلية قبل كتابة أي سطر = `methodology/PRE_IMPLEMENTATION_CONSOLIDATION_REVIEW.md` §7 — تُقرأ نصاً مع كل بطاقة مهمة.

## §8 مخرجات الوكيل
كل تسليم = التغييرات + تقرير فحص ذاتي (نقاء/حدود/نطاق/بوابة) + إدخال Handoff مقترح. **لا commits ولا push ولا وسوم ولا releases من أي وكيل** — كلها بيد SA حصراً.
