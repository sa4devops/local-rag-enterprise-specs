# UI_AI_WORKSPACE_MODEL.md — نموذج مساحة العمل الذكية (ws.main)

> **Version:** 1.1 — (Δ v0.8: +§14 وضعا العرض محادثة/بطاقات — OD-WS-4 مقفل بقرار SA 2026-07-10) · **Date:** 2026-07-10 (الأصل 2026-07-06) · **الموضع الهدف:** `ui/UI_AI_WORKSPACE_MODEL.md`
> **المراجع الحاكمة:** Phase 6 design (chat/Validation/SQL/Reports) · Phase 3 (Action Registry) · Phase 4 (المفسّر/التصنيف) · Phase 5 (No-Guessing/provenance) · A-UX-1..14 · constitution م1/م2/م3.
> **الثابت الأول:** الـ Workspace **لا ينفّذ شيئاً بنفسه** — هو سطح يولّد **اقتراحات** ويعرض **نتائج**؛ التنفيذ حصراً عبر Action Layer الخادمي (نفس مسار الـ Renderer).

## 1) التعريف والتموضع
`ws.main` هو السطح المحادثي المركزي للمستخدم من P6: محادثة مسنودة + بطاقات أفعال + لوحات سياق. ليس chatbot (المحادثة وسيلة تشغيل لا مصدر حقيقة)، وليس بديلاً عن السطح الحتمي (A-UX-1) — الصفحة الرئيسية الافتراضية بعد P6 (علَم لكل دور، OD-UX-1).

## 2) تشريح الشاشة (مناطق ثابتة)
| المنطقة | المكوّنات (معرّفات للمجموعتين G3/G4) |
|---|---|
| الإطار | topbar (بحث موحّد، جرس الإشعارات، مبدّل ar/en، هوية المستخدم) · sidebar مصغّر (Workspace، /me/*، /app حسب الصلاحية) |
| شريط الجلسات | `ws.session_rail` (جلساتي، ربط بمشروع، عنوان تلقائي قابل للتحرير) |
| لوحة المحادثة | `ws.thread` (رسائل، بطاقات، شارة **مُولَّد** على كل مخرج نموذج) |
| المُنشئ | `ws.composer` (نص، إرفاق ملف، **`ws.scope_selector`** لنطاق المعرفة — يعرض المسموح فقط) |
| اللوحة اليمنى (تبويبات) | `panel.citations` · `panel.related_records` · `panel.attachments` · **`panel.audit`** (بصلاحية فقط — §8) |
| لافتات النظام | `banner.model_down` · `banner.degraded` · لافتة Safe Mode إن فُعّل |

## 3) خط النية→البطاقة→التنفيذ (الجوهر — حالات مسماة)
**القاعدة:** مخرج النموذج في مسار الأفعال هو **Action Proposal مهيكل** (action_id + قيم مرشّحة مأخوذة من schema الحقول) — لا يُنفَّذ ولا يُعرض كزر تنفيذ مباشر أبداً.

| # | الحالة | مَن يقرر | ما يظهر للمستخدم |
|---|---|---|---|
| S1 | INTENT — تفسير النية | النموذج (اقتراح) | مؤشر كتابة/بث |
| S2 | ROUTE — سؤال معرفي أم فعل أم استيضاح | النموذج (اقتراح) + قواعد | — |
| S3 | PROPOSED — بطاقة اقتراح فعل | النموذج يملأ من **schema** الكيان (A-UX-4) | `card.action_preview` بحقول قابلة للتحرير يدوياً |
| S4 | CLARIFYING — نقص مدخلات | حتمي: من `required_inputs` والتحققات | أسئلة موجهة **حقلاً حقلاً** أو تحرير مباشر داخل البطاقة — لا أسئلة عامة |
| S5 | VALIDATED — تحقق خادمي | **الخادم** (نفس validators الـ Renderer) | أخطاء مثبتة على الحقول داخل البطاقة |
| S6 | AUTHORIZED — بوابة الصلاحية | **الخادم** (PermissionAPI) | إن رُفض ⇒ §6 |
| S7 | CONFIRMED — تأكيد صريح | المستخدم | زر تأكيد؛ الحرِج = تأكيد مكتوب/مزدوج حسب `risk_level` |
| S8 | PENDING_APPROVAL — عند `approval_required` | مسار الاعتماد (P3/P7 HITL) | شارة «بانتظار الاعتماد» + تتبع؛ **لا تنفيذ قبل قرار المعتمِد** في `queue.approval_detail` |
| S9 | EXECUTING → SUCCEEDED/FAILED | الخادم عبر عملية الـ backend المعلنة | `card.result`: النتيجة + روابط (فتح السجل في `run.detail`) + **مرجع التدقيق** + تراجع إن `rollback_supported` |

**إبطال المعاينة:** البطاقة تحمل نسخة السياق؛ تغيّر السجل/السياسة قبل التنفيذ ⇒ إعادة تحقق حية إلزامية (constitution م16) وتظهر «حدِّث المعاينة».

## 4) مسار السؤال المعرفي (بلا فعل)
استرجاع محكوم (P5) → إجابة بعلامات استشهاد سطرية [1][2] مرتبطة بـ `panel.citations` → **قاعدة «مصدر أو رفض»**: ادعاء أعمال بلا مصدر يُحجب (Validation fail-closed) وتظهر بطاقة `card.refusal_evidence`. الأسئلة البيانية ⇒ `card.sql_answer` (نص SQL ظاهر + جدول ضمن الصلاحية + شرح مستند للصفوف فقط — P6 FR-6.5/6.6).

## 5) عرض الاستشهادات (`panel.citations`)
لكل مصدر: العنوان · المحدد الدقيق (صفحة/مقطع/طابع زمني للوسائط) · شارة التصنيف · وسم المنشأ (مستند / سجل / **نظام: X** لموصلات P7 / OKF-موسوم) · زر **فتح المصدر** (`source.open` — فحص صلاحية لحظة النقر وقد يُرفض بأدب) · نسخ مرجع. فتح مقطع وسائط يقفز للطابع الزمني.

## 6) الرفض عند نقص الصلاحية (ثلاث طبقات — متعمَّدة اللاتماثل)
1. **تشذيب مسبق:** ما لا يملكه المستخدم لا يُقترح أصلاً (اقتراحات الأفعال مرشّحة بقدراته).
2. **رفض فعل صريح:** نية تطابق فعلاً غير مخوّل ⇒ `card.refusal_permission`: «هذا الإجراء يتطلب صلاحية {اسم عام}» **دون كشف نص السياسات** (المفسّر التفصيلي للمخوّلين فقط — P4)، مع فعلين آمنين: «طلب وصول» (إن فُعّل workflow لذلك) و«إنشاء مهمة لمن يملكها».
3. **بيانات مُرشّحة بصمت:** نتائج السجلات تخضع RLS دون الإفصاح بوجود ما لا يراه (اتساقاً مع اختبار P2: «لا يعرف بوجود غيرها») — الصمت للبيانات، والوضوح للأفعال.

## 7) النتائج الكبيرة → Runtime Renderer (Dual-Surface عملياً)
عتبة العرض داخل البطاقة = **10 صفوف** (OD-UX-4): تُعرض خلاصة + أول النتائج + زر **«عرض الكل في القائمة»** يفتح `run.list` **بنفس الفلتر مُسلسلاً** (عقد فلترة من الميتاداتا) — لا ترقيم صفحات داخل الدردشة. التحرير المعقد/الجداول ⇒ زر «فتح النموذج» يفتح `run.form` **مملوءاً مسبقاً** من البطاقة. نفس `action_id` على السطحين دائماً.

## 8) أثر التدقيق للمخوّلين (`panel.audit`)
- **كل مستخدم:** إيصال خفيف على بطاقته: «تم التنفيذ — مرجع تدقيق #…» (رقم مرجعي فقط).
- **المخوّلون (`audit.view` ضمن نطاقهم):** تبويب اللوحة يعرض سلسلة الحدث كاملة (PROPOSED→VALIDATED→AUTHORIZED→CONFIRMED→APPROVED?→EXECUTED) بالفاعل والوقت وrequest/correlation IDs، مرشّحة بنطاق الناظر — نفس محرك `admin.audit` لا محرك ثانٍ.

## 9) أوضاع التدهور (النموذج غير متاح/بطيء) — **العمل لا يتوقف**
| الحالة | المصدر | سلوك الواجهة |
|---|---|---|
| النموذج غير متاح | `/health/llm` | `banner.model_down` + تحوّل المُنشئ إلى **الوضع الحتمي**: بحث كلمات مباشر (OpenSearch — حتمي مسموح) + **مشغّل أفعال** (لوحة أوامر بالأفعال المسموحة تفتح بطاقات/نماذج Renderer) + كل /me/* والـ Renderer تعمل كاملة. **لا اصطفاف للرسائل** (قابلية تنبؤ) — رسالة واضحة بإعادة المحاولة |
| بطء/انقطاع بث | مهلات الـ Gateway | بث بنبض؛ تجاوز المهلة الناعمة ⇒ خيار إعادة/إلغاء؛ تحويل fallback يُعلن بشارة «نموذج بديل» (P6 FR-6.1) |
| عملية طويلة (تقرير/فهرسة) | طبيعة الفعل | تتحول **مهمة غير متزامنة** فوراً: بطاقة حالة + إشعار عند الاكتمال — لا حجز للمحادثة |
| Safe Mode | P4 | لافتة + تقييد القدرات لغير A10 مع رسائل مفسَّرة |

## 10) قواعد الجلسة والسياق
الجلسة ملك صاحبها (لا سياق عابر للمستخدمين) · ربط اختياري بمشروع (تظهر في `projects.main`) · `ws.scope_selector` يضيّق ولا يوسّع أبداً فوق الصلاحية · احتفاظ الجلسات مدة من الإعداد (OD-WS-2) · المرفق المرفوع يدخل خط P5 بوراثة تصنيف السياق ويظهر في `panel.attachments` بحالته.

## 11) كتالوج الحالات الأولى (تُصمَّم كلها في G3/G4 — ليست رسائل خطأ عامة)
فارغة أولى (إرشاد قدرات حسب الصلاحية) · بث/تفكير · `card.action_preview` (+أخطاء حقول) · CLARIFYING · PENDING_APPROVAL · EXECUTING · `card.result` نجاح · فشل تنفيذي (error_code + request_id + «إبلاغ») · `card.refusal_permission` · `card.refusal_evidence` (No-Guessing: ما بُحث، ولماذا لا يكفي، وأفعال آمنة) · `card.sql_answer` · `banner.model_down` · وضع حتمي نشط.

## 12) RTL وإتاحة
انعكاس كامل منطقي (البطاقات/اللوحات start/end) · تنقّل لوحة مفاتيح إلى داخل البطاقات وأزرارها · قارئ شاشة يقرأ حالة البطاقة (S-state) · تباين شارات التصنيف وفق نظام التصميم (دفعة 3).

## 14) وضعا العرض لـ ws.main: محادثة / بطاقات — (Δ v0.8 · OD-WS-4 · مقفل بقرار SA 2026-07-10)

**القاعدة:** لمنطقة `ws.thread` وضعا عرض يختارهما المستخدم كـ **تفضيل مستخدم** (user preference — مفتاح واجهي `ws.view_mode` بقيمتي `cards` | `conversation`؛ التبديل فوري، يُحفظ لكل مستخدم، ولا يغيّر أي سلوك خادمي):

| البعد | وضع البطاقات (القائم — الافتراضي الحالي) | وضع المحادثة (المضاف) |
|---|---|---|
| بنية العمود | thread ببطاقات مؤطَّرة (card.action_preview / card.result …) | **عمود محادثة نظيف**: صفوف رسائل متتالية بفواصل رفيعة، بلا أُطر بطاقات حول النص |
| رد المساعد | داخل بطاقة/إطار | **نص كامل العرض** (full-width) داخل عمود المحادثة، بشارة «مُولَّد» القائمة |
| الأفعال | بطاقة معاينة كاملة | **أفعال inline مضغوطة** بذيل الرسالة — عرضٌ لنفس عقد الأزرار الـ13 حرفياً (متغيّر «inline message action» — UI_ACTION_BUTTON_MODEL §2.1)؛ النقر يفتح المعاينة/التأكيد بنفس الخط S3–S9 دون أي اختصار |
| المرفقات والملفات | panel.attachments + بطاقات | **مرفقات ورفع ملفات داخل الرسائل** + **تنزيل الملفات المولّدة من داخل الرسالة** — نفس خط P5 (file.ingest بوراثة التصنيف) ونفس أفعال الفتح المحكومة؛ لا قناة رفع/تنزيل جديدة |
| المصادر (الاستشهادات) | panel.citations يفتح تلقائياً عند حدث | **مطوية افتراضياً**: سطر مصادر مضغوط أسفل الرد (عدد + توسيع inline)؛ panel.citations يبقى متاحاً عند الطلب ولا يفتح تلقائياً في هذا الوضع |

**ثوابت لا يغيّرها الوضعان (قائمة قطعية):** خط النية→البطاقة→التنفيذ S1–S9 كما هو (§3) · نفس action_id وعقد الحقول الـ13+ والتقييم الخادمي · الرفض بطبقاته الثلاث (§6) · قاعدة «مصدر أو رفض» (§4 — طيّ المصادر عرضيّ ولا يُسقطها) · عتبة الـ10 صفوف والتحويل إلى الـ Renderer (§7) · panel.audit وأحكامه (§8) · أوضاع التدهور (§9) · شارة «مُولَّد» على كل مخرج نموذج · قيود D9. **الوضع عرضيّ (presentational) صرف** — أي حاجة مستقبلية لاختلاف سلوكي بين الوضعين تتطلب قراراً جديداً من SA.
