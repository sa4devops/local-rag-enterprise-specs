# UI_VISIBILITY_RULES.md — قواعد الظهور والإخفاء

> **Version:** 1.0 — Proposed (v0.5) · **Date:** 2026-07-07 · **الموضع الهدف:** `ui/UI_VISIBILITY_RULES.md`
> **مبنيّ فوق:** UI_UX_ASSUMPTIONS (A-UX-6/7/8/10/11) · UI_ACTION_BUTTON_MODEL (visible_when/disabled_when) · phases/designs (P1..P8).
> **المبدأ:** الظهور قرار **متعدد الأبعاد** يُقيَّم في الخادم أولاً، والواجهة **مرآة**. المخفي بأمر السياسة لا يصل للمتصفح أصلاً؛ المعطَّل يظهر بسبب مقتضب؛ المقنَّع لا يغادر الخادم.

## 1) درجات الرؤية الأربع (تُطبَّق بترتيب صارم)
| الدرجة | التعريف | مَن يقرر | ما يصل للمتصفح |
|---|---|---|---|
| **HIDDEN-SERVER** | لا يُرسَل من الأصل | الخادم (الصلاحية/التصنيف/العلَم) | لا شيء (لا DOM، لا شبح) |
| **HIDDEN-CLIENT** | يُرسَل لكن مطويّ للسياق (Progressive Disclosure) | الواجهة وفق قواعد §5 | كامل، لكن مطويّ |
| **DISABLED-WITH-REASON** | مرئي مع قفل + سبب i18n | الخادم يقرّر الحالة، الواجهة تعرض | كامل + سبب مقتضب |
| **VISIBLE** | ظاهر تفاعلياً | التقييم الكامل نجح | كامل نشط |

**قاعدة الأمن الحاسمة:** أي قرار حساس (كشف حقل Secret، إظهار زر critical) **HIDDEN-SERVER** حصراً — لا يجوز أن يعتمد على منطق الواجهة. الواجهة قد تُخفي إضافياً للترتيب، لكنها لا تُظهر ما لم يرسله الخادم.

> **قاعدة D9 البنيوية (Δ v0.6):** الأسرار وendpoints الحقيقية (Base URL/API Key/tokens) **HIDDEN-SERVER دائماً وبنيوياً** — لا تُخزَّن في طبقة الواجهة ولا تصل إليها في أي حالة من الحالات الخمس عشرة؛ `secret_ref` يظهر **بالاسم فقط** (تقنيع بنيوي S10). وحالة التشغيل `paused_awaiting_approval` تُعامل وسمياً كـ `after-approval` على مستوى الـ run — البند يُقرَّر على سطح الاعتماد الرسمي حصراً.

## 2) الأبعاد التي تحكم القرار (خمسة مصادر تدخل التقييم الخادمي)
1. **Permission** (RBAC + ABAC + ReBAC + NtK) — البعد الأولي.
2. **Classification** (5 درجات + وسم NtK) — يشذّب الحقول والصفوف والأزرار.
3. **Risk-level للفعل** (low→critical) — يقود إظهار طبقات التأكيد/الاعتماد.
4. **Workflow state** — حالة السجل الحالية تُبيح/تمنع الأزرار (transition ثم effect).
5. **Feature flag / Safe Mode** — العلَم الرئيسي، وSafe Mode يقيّد إلى A10 + قراءة.

يُضاف بُعد رقابي: **Audit scope** — بعض المحتوى (سلسلة correlation، سجلات تدقيق) يظهر فقط لمن يملك `audit.view` ضمن نطاقه.

## 3) قواميس الظهور (فئات ملزمة — تُلصق كوسم على كل عنصر في الشاشات)
| الوسم | يعني | مثال تطبيقي |
|---|---|---|
| `always` | ظاهر دائماً بلا شرط | topbar الشعار، مبدّل ar/en، رقم الإصدار في about |
| `on-demand` | ظاهر عنوانياً/مطويّ، يفتحه المستخدم | panel.citations رأسه ظاهر، محتواه مطويّ |
| `by-permission` | ظاهر إذا تحققت صلاحية معلنة | زر record.delete ⇒ `records.{e}.delete` |
| `hidden-by-permission` | HIDDEN-SERVER لغير المخوّل | زر migration.approve لغير المعتمِد |
| `by-classification` | يظهر/يُقنَّع حسب تصنيف العنصر مقابل clearance الناظر | حقل رقم قومي (Secret) ⇒ «•••» + قفل لغير المخوّل |
| `admin-only` | يظهر فقط في Console وفق دور admin | admin.record_types، Policy Studio |
| `superadmin-only` | ضمن admin-only لكن يقتصر على SuperAdmin | admin.policies، admin.capabilities |
| `ops-only` | يظهر لدور Ops فقط (قراءة) | ops.status، admin.config_profiles |
| `after-validation` | يظهر بعد نجاح Validation الخادمية | زر Confirm في `card.action_preview` |
| `after-approval` | يظهر بعد قرار اعتماد | زر «تنفيذ» في بند اعتماد + شارة Approved |
| `audit-visible` | يظهر للمخوّل بـ `audit.view` ضمن نطاقه فقط | panel.audit، مرجع تدقيق مفصَّل |
| `debug-visible` | يظهر في وضع تشخيص خادمي مفعَّل | تفاصيل error stack — **معطَّل افتراضياً في الإنتاج** |
| `flag-gated` | مرتبط بـ feature flag/A10 | ميزة قيد التجربة ⇒ ظاهرة لأدوار الاختبار |
| `safe-mode-gated` | يُخفى/يُعطَّل في Safe Mode | أي فعل خارج A10 وقراءة |
| `state-gated` | مربوط بحالة workflow معينة | زر «إرجاع» يظهر فقط في حالات approvable |

## 4) مصفوفة التقييم الملزمة (الترتيب لا يُغيَّر)
1. **Feature/Safe Mode** — علَم مغلق أو Safe Mode ⇒ HIDDEN-SERVER (أو DISABLED للمخوّلين حسب السياسة).
2. **Permission (RBAC/ABAC/ReBAC)** — فشل ⇒ HIDDEN-SERVER.
3. **NtK/Classification** — الناظر أقل من التصنيف ⇒ HIDDEN-SERVER للسطر/المقنَّع للحقل الحساس.
4. **Workflow state** — حالة السجل لا تسمح ⇒ DISABLED-WITH-REASON («الحالة الحالية لا تسمح بهذا الفعل»).
5. **Risk + policy** — يقرّر طبقات التأكيد والاعتماد (تُعرض `after-validation` / `after-approval`).
6. **Progressive Disclosure** — إذا مرّ كل ما سبق، تقرّر الواجهة: `always` / `on-demand` (§5 في `UI_PROGRESSIVE_DISCLOSURE.md`).

**قاعدة الرفض:** أي «سماح» يتطلب مرور كل الأبعاد؛ أي «منع» يقطع السلسلة فوراً.

## 5) قواعد لكل بُعد (تفصيلاً)

### 5.1 Permission (RBAC + ABAC + ReBAC + NtK)
- زر بلا صلاحية ⇒ **hidden-by-permission** (HIDDEN-SERVER)، ليس disabled.
- قائمة بلا صلاحية `read` على أي عنصر ⇒ الصفحة نفسها لا تُقدَّم (redirect إلى دخول مصرَّح أو 403 مفسَّرة).
- علاقة ReBAC منتهية ⇒ الصلاحية تُلغى تلقائياً، والعنصر يختفي في التنقّل التالي.
- الرفض في ws.main ⇒ `card.refusal_permission` بصياغة عامة («يتطلب صلاحية {اسم عام}») — **لا كشف نص السياسة**.

### 5.2 Classification (خماسي)
- الحقل ذو تصنيف > clearance الناظر: **مقنَّع** «•••» + قفل + سبب عام. لا يُرسل الأصل.
- الصف ذو تصنيف > clearance: **HIDDEN-SERVER** — لا يظهر في العدّ ولا الترقيم (اتساقاً مع P2: «لا يعرف بوجود غيرها»).
- الملف والمشتقات (نص/chunks/إطارات) ترث الأشد.
- التصدير: كل صف/حقل يُفحَص قبل التضمين — الرفض في الملف ذاته لا في مرحلة العرض.

### 5.3 Risk-level (low→critical)
- low: زر بلا تأكيد، ظاهر مباشرة عند توفر الصلاحية.
- medium: تأكيد simple (inline).
- high: تأكيد **typed** في modal + اعتماد حسب السياسة.
- critical: **dual** (زر ثانٍ يتفعل بعد عدّاد قصير) + HITL/اعتماد إلزامي + شارة «يتطلب اعتماداً في الإنتاج» **ظاهرة قبل النقر**.

### 5.4 Workflow state
- كل زر transition/فعل مربوط بحالة السجل: خارج الحالة ⇒ **DISABLED-WITH-REASON**.
- عرض شريط الحالة أعلى `run.detail` بشارة الحالة الحالية.
- زر «إرجاع» في اعتماد يظهر فقط بعد submit ولا يظهر بعد Approved.
- التغيير من الخادم يعيد تقييم كل الأزرار فورياً (م2 + م16).

### 5.5 Feature flags + Safe Mode
- flag=off ⇒ العنصر/المسار **HIDDEN-SERVER**.
- Safe Mode ⇒ كل ما ليس A10 وقراءة يُصبح DISABLED-WITH-REASON عام: «الوضع الآمن مفعَّل».
- A10 لا يمكن تعطيلها أصلاً؛ محاولة العرض «تعطيل» ترفض 403 حتى للسوبر أدمن.
- flag «قيد التجربة» (canary) ⇒ ظاهر لأدوار الاختبار فقط، ويعرض شارة «تجريبي».

### 5.6 Audit visibility
- `panel.audit` في ws.main: **رأس التبويب مخفي كلياً** لغير المخوّل (لا شيء يخبره بوجوده).
- للمخوّل: مطويّ افتراضياً، محتواه مرشَّح بنطاقه.
- سلسلة correlation في `admin.audit` **بنطاق الناظر** لا كامل النظام.
- **مرجع التدقيق #ID** في بطاقة نتيجة يظهر لصاحب البطاقة كإيصال (لا كتفصيل).

### 5.7 Debug
- تفاصيل stack/errors لا تُعرض في الإنتاج بلا وضع تشخيص خادمي (config Ops).
- في dev/staging: يظهر لمجموعة أدوار محدّدة فقط بشارة «تشخيص».

## 6) قواعد التقنيع vs الإخفاء (متى أيّهما؟)
| الحالة | الحكم |
|---|---|
| الصف كله خارج نطاق الناظر | **إخفاء تام** (HIDDEN-SERVER) بلا عدّ |
| صف مسموح لكن يحوي حقولاً حساسة | **إخفاء الصف مسموح للعرض؛ الحقل الحساس مقنَّع** |
| زر خارج الصلاحية | **إخفاء تام** (لا disabled) |
| زر ضمن الصلاحية لكن الحالة لا تسمح | **DISABLED-WITH-REASON** |
| نتيجة استرجاع خارج الصلاحية | **إخفاء تام قبل الإرجاع** (فلترة استعلامية P5) |

## 7) قواعد شفافية «لماذا مخفي/معطَّل؟»
- **الفعل** (action): إظهار سبب مقتضب دائماً — «يتطلب صلاحية X» / «الحالة الحالية Y» — **دون كشف نص السياسة**.
- **البيانات** (rows/records): **صمت** — عدم الإفصاح بوجود ما لا يراه (لاتماثل مقصود، P2).
- **الحقل المقنَّع**: أيقونة قفل + tooltip عام «حقل حساس — يتطلب تصريحاً أعلى».
- المفسّر التفصيلي متاح للمخوّلين في `admin.permissions` عبر لوحة المحاكاة.

## 8) مثال تطبيقي شامل (لعنصر واحد: زر `record.delete` في `run.detail`)
| البُعد | القاعدة | النتيجة |
|---|---|---|
| Safe Mode مفعَّل | خارج A10 | DISABLED-WITH-REASON: «الوضع الآمن مفعَّل» |
| Permission (`records.{e}.delete`) | غير متوفرة | HIDDEN-SERVER (لا زر أصلاً) |
| Classification (السجل Secret، الناظر Confidential) | الناظر أدنى | الصف كله HIDDEN — لا نصل حتى للزر |
| Workflow state (السجل «مُعتمَد» ولا يقبل الحذف) | لا يسمح | DISABLED-WITH-REASON: «السجل مُعتمَد — لا يمكن حذفه» |
| Risk=high | يستوجب typed + approval | الزر ظاهر ⇒ ضغط ⇒ modal typed ⇒ بند اعتماد |
| Audit | كل خطوة | RECORD_DELETED عند التنفيذ، وسلسلة correlation في `admin.audit` |

## 9) قواعد Stitch الملزمة (تُلصق في كل prompt يعرض عناصر ذات صلاحية)
1. **لا تخترع أزراراً بلا صلاحية مذكورة** — كل زر يجب أن يكون له `permission` واضح أو `always`.
2. **لا تعرض حقولاً حساسة بقيم فعلية** — القيم دائماً «•••» في حالات المعاينة الحساسة.
3. **لا رأس تبويب لـ `panel.audit`** في تصاميم مستخدم عادي (بدون audit.view).
4. **عرض السبب بجانب disabled**: كل زر معطَّل يجب أن يظهر بجانبه tooltip بسبب مقتضب.
5. **عرض شارة الحالة** أعلى `run.detail` دائماً (تصنيف + workflow state + مرجع تدقيق قابل للنقر للمخوّل).
