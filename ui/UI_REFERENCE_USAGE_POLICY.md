# UI_REFERENCE_USAGE_POLICY.md — سياسة استخدام المراجع والصور في التصميم

> **Version:** 1.0 — Proposed · **Date:** 2026-07-06 · **الموضع الهدف:** `ui/UI_REFERENCE_USAGE_POLICY.md`
> **السلطة:** امتداد **بصري/تجريبي** لسياسة Reference-aware Clean Implementation (D8/ADR-0021) وكتالوج `references/OSS_REFERENCE_CATALOG.md` — عند التعارض تحكم وثائق v0.3.
> **النطاق:** مُلزمة لكل جلسة Google Stitch / Figma / أي مولّد واجهات، ولكل مصمم بشري أو وكيل ذكاء يعمل على واجهات هذا النظام.

## §1 المبدأ: مرجعان مختلفان بقاعدتين مختلفتين
- **مرجع كود:** يحكمه كتالوج OSS وسياسة D8 حصراً (عتبة ~10 أسطر، بوابة الترخيص، حفظ النسب) — خارج نطاق هذا الملف.
- **مرجع بصري/تجريبي (هذا الملف):** صور وتدفقات **للإلهام فقط** — أي فهم الأنماط (بنية المعلومات، التدفق، الكثافة، مواضع الحوكمة) ثم **إعادة تعبير أصلية** بلغة «قيادة مؤسسية هادئة». الإلهام يُنتج فهماً؛ لا يُنتج نسخاً.

## §2 المسموح
دراسة لقطات وفيديوهات عامة لأنظمة مؤسسية · استخلاص أنماط عامة (list+detail، صناديق اعتماد، لوحات محاكاة، مراقبات pipelines) · مقارنة قرارات UX وتدوين الدروس · إرفاق صور مرجعية داخل جلسة Stitch **وفق §4 و§7** · تسمية المرجع صراحة في design-notes.

## §3 الممنوع قطعاً
نسخ أو تقليد **branding / logos / wordmarks / شخصيات / مجموعات أيقونات مميزة** · استنساخ **layout حرفي** أو pixel-level أو تسلسل شاشات مميز كامل · نسخ نصوص واجهة (microcopy) · استخدام لوحة ألوان مميزة لمنتج (لوحتنا في UI_DESIGN_SYSTEM هي المصدر الوحيد) · **نسخ أي كود** من أي مصدر إلى الريبوهات (يشمل كود مخرجات الأدوات) · تضمين لقطات منتجات خارجية داخل ملفات الحزمة أو الريبو.

## §4 قاعدة الأداة الخارجية (حرجة — أمن معلومات)
Stitch وأمثاله **خدمات خارجية**. يُمنع منعاً باتاً إدخال: أسماء الجهة الحقيقية، بيانات موظفين/معاملات، هياكل تنظيمية فعلية، لقطات أنظمة داخلية، أو أي مادة مصنّفة — **بيانات وهمية فقط** (كما تفرضه ديباجة الموجّهات §1). خرق هذه القاعدة يُعامل حادثة أمن معلومات لا مخالفة تصميم.

## §5 مصير مخرجات Stitch
- **الكود المولَّد:** لا يدخل `local-rag-enterprise-platform` ولا `local-rag-enterprise-specs` **إطلاقاً** — prototype/reference only؛ التنفيذ الفعلي يُكتب من الصفر وفق D1.
- **الصور/PDF المصدَّرة:** يجوز حفظها لاحقاً في specs تحت `ui/prototypes/` بأسماء ASCII مع سطر مصدر (المجموعة Gx + التاريخ + الجلسة).
- لا يُستشهد بمخرجات Stitch كمصدر معماري — المصدر هو ملفات `ui/` هذه.

## §6 design-notes إلزامية لكل جلسة
الوجهة المستقبلية `ui/design-notes/` (نمط الاسم: `design-notes-g{n}-YYYY-MM-DD.md`). الحقول الإلزامية: المجموعة والتاريخ · المراجع المستخدمة (أسماء/روابط) · ما تُعلِّم من كل مرجع · **ما خُولف عمداً ولماذا** · ملاحظات للتنفيذ اللاحق. **جلسة بلا design-notes = مخرجاتها غير معتمدة.**

## §7 جدول المراجع المتوقعة (ما يجوز / ما يُمنع)
| المرجع | يجوز | يُمنع |
|---|---|---|
| Open WebUI | تدفقات إدارة الجلسات/الإعدادات **كصور** (وهو مدرج في كتالوج الكود لأغراض الكود بقواعد D8 المنفصلة) | فقاعات الدردشة الاستهلاكية، الهوية، نقل layout حرفي |
| Appsmith / Budibase / Directus | أنماط console وlist+detail والجداول **كصور** | مظهر «builder للمستخدم النهائي»، الأيقونات، أي كود إلا عبر كتالوج الكود |
| Dify / NocoDB | **سلوك عبر لقطات فقط** (قيد v0.2 القائم: بلا قراءة كود يبقى سارياً) | أي اطلاع على الشيفرة، استنساخ تدفق مميز كامل |
| Microsoft Admin Center / Linear / GitHub | تنظيم بصري عام (كثافة، تسلسل هرمي، أنماط قوائم) **كصور** — ليست في كتالوج الكود أصلاً | أي كود/أيقونات/هوية/استنساخ layout |
| أنظمة الجهة الداخلية | فهم الاحتياج من المعرفة التنظيمية | **رفع لقطاتها أو بياناتها لأي أداة خارجية (§4)** |
أي مرجع جديد غير مدرج: يُعامل بالقواعد العامة أعلاه ويُوثَّق في design-notes؛ وإن كان مرجع **كود** فيُضاف لكتالوج OSS أولاً.

## §8 التنفيذ النهائي
حصراً وفق **D1**: React + Vite + TypeScript + Tailwind + shadcn/ui-style، برموز UI_DESIGN_SYSTEM. أي خط/أيقونات/مكتبة UI إضافية ⇒ **صف license-review قبل الاستخدام** (مثال قائم: الخط — OD-DS-2).

## §9 الإنفاذ
بند القسم G في `UI_GATE_REVIEW_CHECKLIST.md` · إقرار الالتزام في handoff كل جلسة/مهمة · مراجعة عينية للتشابه قبل اعتماد مخرجات أي مجموعة · **عند الشك يُعامل العنصر منسوخاً حتى تثبت أصالته**.
